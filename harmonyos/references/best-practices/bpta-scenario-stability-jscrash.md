## Cannot read property xxx of undefined类案例

### 概述

此类问题在实际应用开发调试运行过程中是常见的JS Crash类型，其表示为变量不是预期类型，在代码层面则为对变量的使用未进行事先的校验，在错误日志中报错多表现为如下：

```
1. Error name:TypeError
2. Error message:Cannot read property xxx of undefined
```

### 问题现象

异常JS Crash日志示例：

```
1. Generated by HiviewDFX@OpenHarmony
2. ================================================================
3. Device info:xxxx
4. Build info:xxxx
5. Fingerprint:9851196f9fed7fd818170303296ae7a5767c9ab11f38fd8b0072f0e32c42ea39
6. Module name:com.xxx.xxx
7. Version:1.0.0.29
8. VersionCode:10000029
9. PreInstalled:Yes
10. Foreground:No
11. Pid:2780
12. Uid:20020018
13. Reason:TypeError
14. Error name:TypeError
15. Error message:Cannot read property needRenderTranslate of undefined
16. Stacktrace:
17. Cannot get SourceMap info, dump raw stack:
18. at updateGestureValue (phone/src/main/ets/SceneBoard/recent/scenepanel/recentpanel/RecentGesture.ts:51:51)
19. at onRecentGestureActionBegin (phone/src/main/ets/SceneBoard/scenemanager/SCBScenePanel.ts:5609:5609)
20. at anonymous (phone/src/main/ets/SceneBoard/scenemanager/SCBScenePanel.ts:555:555)
21. at anonymous (phone/src/main/ets/SceneBoard/recent/RecentEventView.ts:183:183)
```

### 分析步骤

1. 提取日志关键信息
   * 通过日志信息可以确定为Type Error类问题，由异常信息得知是在读取needRenderTranslate对象时报错，该对象为undefined。
   * 可以通过异常代码调用栈，获取错误产生位置。
   * Cannot get SourceMap info, dump raw stack信息表示该应用为release包安装，JS栈转换ETS行列号失败，可考虑使用[堆栈轨迹分析](../harmonyos-guides/ide-release-app-stack-analysis.md)来解析行号。
2. 定位到具体代码

   通过以上报错堆栈和报错信息，能够定位到具体代码示例如下：

   ```
   1. // Update the attributes related to manual effects
   2. public updateGestureValue(screenWidth: number, recentScale: number,
   3. sceneContainerSessionList: SCBSceneContainerSession[]): void {
   4. // Calculation of the distance moved by the hand
   5. this.translationUpY =
   6. (this.multiCardsNum >= 1) ? sceneContainerSessionList[this.multiCardsNum - 1].needRenderTranslate.translateY :
   7. 0; // Report an incorrect line number
   8. this.translationDownY =
   9. (this.multiCardsNum >= 2) ? sceneContainerSessionList[this.multiCardsNum - 2].needRenderTranslate.translateY : 0;
   10. this.screenWidth = this.getUIContext().px2vp(screenWidth);
   11. this.recentScale = recentScale;
   12. }
   ```

   [CannotReadPropertyXXXOfUndefinedCase.ets](https://gitcode.com/HarmonyOS_Samples/BestPracticeSnippets/blob/master/JSCrash/entry/src/main/ets/pages/CannotReadPropertyXXXOfUndefinedCase.ets#L27-L39)

### 修复方法

上述分析已经明确为sceneContainerSessionList的成员变量needRenderTranslate在使用的过程中可能存在为undefined的情况。对于这类问题，增加保护性判断可直接规避。具体修改方法可以如下，在访问对象前增加“?”操作符进行判断保护。

```

2. public updateGestureValue(screenWidth: number, recentScale: number,
3. sceneContainerSessionList: SCBSceneContainerSession[]): void {
4. // Calculation of the distance moved by the hand
5. this.translationUpY = (this.multiCardsNum >= 1) ?
6. sceneContainerSessionList[this.multiCardsNum - 1]?.needRenderTranslate.translateY : 0;
7. this.translationDownY = (this.multiCardsNum >= 2) ?
8. sceneContainerSessionList[this.multiCardsNum - 2]?.needRenderTranslate.translateY : 0;
9. this.screenWidth = this.getUIContext().px2vp(screenWidth);
10. this.recentScale = recentScale;
11. }
```

### 建议与总结

对于这类问题，需要开发者在编码阶段充分考虑对象访问的安全性，确保必要的判空处理。同时很多场景单纯的判空可能只能是规避问题，还需要结合业务看看是否是原始对象构造或赋值的逻辑存在问题。

## XXX is not callable类案例

### 概述

表示应用代码试图像调用函数一样去调用某个对象，但该对象不可被调用（不是一个函数或方法），xxx表示不可被调用对象的值。

* 误将变量名用作函数名
* 对象方法不存在或未正确引用

该问题主要场景：

* 类被混淆了，开发者调用对应的方法报出的问题。（混淆导致的方法不存在）
* 开发者调用的对象出现了非预期的值或者该对象无相关方法，例如number没有trim方法。
* 开发者自定义的napi，so未加载，调用对应方法不存在而报错。

### 问题现象

异常JS Crash日志示例：

```
1. Device info:xxx
2. Build info:xxx
3. Fingerprint:654897efdb96b3520481880c8ed746cb2e76916639456c47a8513a3575b3424b
4. Timestamp:xxx
5. Module name:com.example.myapplication
6. Version:1.0.0
7. VersionCode:1000000
8. PreInstalled:No
9. Foreground:Yes
10. Pid:50237
11. Uid:20020199
12. Process Memory(kB): 126783(Rss)
13. Device Memory(kB): Total 11679172, Free 414280, Available 4485120
14. Reason:TypeError
15. Error name:TypeError
16. Error message:undefined is not callable
17. Stacktrace:
18. Cannot get SourceMap info, dump raw stack:
19. at b entry (entry/src/main/ets/pages/test.js:18:5)
20. at test1 entry (entry/src/main/ets/pages/test.js:15:5)
21. at anonymous entry (entry/src/main/ets/pages/Index.ts:18:11)
22. HiLog:
23. ...
```

### 分析步骤

通过以上报错堆栈和报错信息，能够定位到具体代码示例如下：

```
1. function b(a) {
2. a.trim(); // 报错undefined is not callable;
3. }

5. export function c() {
6. let a = 123;
7. b(a);
8. }
```

[XXXIsNotCallableCase.js](https://gitcode.com/HarmonyOS_Samples/BestPracticeSnippets/blob/master/JSCrash/entry/src/main/ets/pages/XXXIsNotCallableCase.js#L6-L14)

原因分析：入参是number类型，number类型无trim函数。

### 修复方法

这个问题在于b(a)调用时传递的是数字123，但b函数中却试图调用字符串方法trim()，数字没有trim方法，所以传参前将其转换成字符串即可。

```
1. function b(a) {
2. a.trim();
3. }

5. export function c1() {
6. let a = 123;
7. b(a.toString()); // 转换为字符串
8. }
```

[XXXIsNotCallableFix.js](https://gitcode.com/HarmonyOS_Samples/BestPracticeSnippets/blob/master/JSCrash/entry/src/main/ets/pages/XXXIsNotCallableFix.js#L6-L14)

### 建议与总结

* 检查错误位置：查看错误堆栈确定哪一行代码触发了错误。(如果开启混淆功能下的代码压缩功能，需先还原报错代码栈，具体操作可参考：[报错栈还原](../harmonyos-guides-V14/source-obfuscation-V14.md#报错栈还原))。
* 检查对象类型：使用typeof操作符来检测数据类型。
* 设置参数默认值：例如为参数设置空字符串默认值，确保始终有字符串可供操作。

## taskpool：failed to serialize result类问题案例

### 概述

taskpool执行任务失败，返回了序列化失败的异常信息。这是因为通信对象是不可序列化的类型。

### 问题现象

由于taskpool执行任务失败不会产生JS Crash，所以只能观察hilog日志。而在taskpool序列化结果失败场景下，其会在hilog日志中抛出异常failed to serialize result。

```
1. //在hilog日志中查找
2. resultError task catch message:An exception occurred during serialization, taskpool: failed to serialize result.
3. Serialize error: Serialize don't support object type: Function
```

在catch到的信息里，前一段信息里有“during serialization”，表明是在序列化中。后一段是“failed to serialize result”，表明是返回结果的序列化失败，需要看下返回的结果是什么类型。

### 分析步骤

```
1. @Concurrent
2. export function printArgs(args: number): number {
3. return args
4. }
```

上面的例子是在执行的任务中创建了一个task任务，并且返回。但是这个createTask函数返回值task对象是无法被序列化的，从而导致了任务的执行失败。

## XXX is not initialized类问题案例

### 概述

变量未被初始化发生的原因通常是模块间循环依赖导致的。

### 问题现象

异常JS Crash日志示例：

```
1. Device info:xxx
2. Build info:xxx
3. Fingerprint:xxx
4. Timestamp:xxx
5. Module name:xxx
6. Version:1.1.10.210
7. VersionCode:101010210
8. PreInstalled:Yes
9. Foreground:No
10. Pid:16128
11. Uid:20020026
12. Reason:ReferenceError
13. Error name:ReferenceError
14. Error message:z822 is not initialized
15. Stacktrace:
16. Cannot get SourceMap info, dump raw stack:
17. at static_initializer (entry|entry|1.0.0|src/main/ets/engine/en/KbdEnLayoutUtils.ts:1:1)
18. at func_main_0 (entry|entry|1.0.0|src/main/ets/engine/en/KbdEnLayoutUtils.ts:0:3016)
19. HiLog:
20. ...
```

### 分析步骤

报错xxx is not initialized不一定是循环依赖问题，需要通过hdc命令打开循环依赖检测开关进一步确认，无需重启设备。开关命令如下：

```
1. 开启：hdc shell param set persist.ark.properties 0x2000105c
2. 关闭：hdc shell param set persist.ark.properties 0x0000105c
```

开关打开后复现报错就是第一现场报错，解决即可。如果报错仍是xxx is not initialized，就是文件存在循环依赖。

循环依赖原理：

* 模块加载顺序：根据ECMA规范，会按照深度优先的顺序进行加载。假设应用存在加载链路A->B->C，那么它的执行顺序则为C->B->A。
* 循环依赖：如果应用存在加载链路A->B->A，根据深度遍历执行顺序，执行流程会先标记A的状态为加载中，然后去加载B，标记B的状态为加载中，然后去加载A，由于A文件已经标记加载中，根据规范定义，识别到加载中模块会直接返回，就会先执行B文件。

部分循环依赖会产生crash的原因：

由上面的叙述可知，B文件依赖A文件，但是B文件先执行，如果B文件导入的A文件变量没有在全局或者静态类中被使用，B文件就会正常执行。如果B文件变量在全局或者静态类中被使用，导致文件执行时就会用到A的变量，就会产生xxx is not initialized的crash，即循环依赖导致变量未被初始化。

反例：

```
1. import { Animal } from './XXXIsNotInitializedCase1'

3. export let a = "this is A";

5. export function A() {
6. return new Animal;
7. }
```

[XXXIsNotInitializedCase.ets](https://gitcode.com/HarmonyOS_Samples/BestPracticeSnippets/blob/master/JSCrash/entry/src/main/ets/pages/XXXIsNotInitializedCase.ets#L6-L13)

```
1. import { a } from './XXXIsNotInitializedCase'

3. export class Animal {
4. static {
5. console.log("this is in class");
6. let str = a; // 报错信息：a is not initialized
7. }
8. }
```

[XXXIsNotInitializedCase1.ets](https://gitcode.com/HarmonyOS_Samples/BestPracticeSnippets/blob/master/JSCrash/entry/src/main/ets/pages/XXXIsNotInitializedCase1.ets#L6-L14)

正例:

```
1. import { Animal } from './XXXIsNotInitializedFix1'

3. export let a = "this is A";

5. export function A() {
6. return new Animal;
7. }
```

[XXXIsNotInitializedFix.ets](https://gitcode.com/HarmonyOS_Samples/BestPracticeSnippets/blob/master/JSCrash/entry/src/main/ets/pages/XXXIsNotInitializedFix.ets#L6-L13)

```
3. export class Animal {
4. static {
5. console.log("this is in class");
6. }
7. str = a; // 修改点
8. }
```

[XXXIsNotInitializedFix1.ets](https://gitcode.com/HarmonyOS_Samples/BestPracticeSnippets/blob/master/JSCrash/entry/src/main/ets/pages/XXXIsNotInitializedFix1.ets#L6-L14)

### 修复方法

循环依赖的检测方法：安全规则[@security/no-cycle](../harmonyos-guides/ide_no-cycle.md)。

### 建议与总结

循环依赖的根本原因是模块/类之间的职责边界不清晰、耦合度过高。因此，最佳的解决方式是预防优于治疗，通过良好的设计来避免其产生。

## Error类案例

### 概述

Error类问题一般是开发者或JS库主动抛出来的JS异常。

这类问题目前有两种场景：

1. 应用程序遇到无法解决的故障，需要考虑抛出JS异常来终止业务并生成故障日志。
2. 依赖使用JS库或者其他模块的接口，若可能抛出异常，需要考虑使用try-catch机制进行捕获，否则会终止当前业务。

### 问题现象一

开发者自主抛出JS异常，可通过如下代码实现：

```
1. // In a production environment, encrypted error codes should be used instead of plain text descriptions.
2. throw new Error("TEST JS ERROR")
```

[ThrowNewError.ets](https://gitcode.com/HarmonyOS_Samples/BestPracticeSnippets/blob/master/JSCrash/entry/src/main/ets/pages/ThrowNewError.ets#L25-L26)

通过DevEco Studio的FaultLog工具可收集到此类异常日志，其中JS异常栈信息可直接定位到抛异常的代码位置。

### 问题现象二

因未处理三方接口抛出的JS异常导致JS Crash日志示例：

```
1. Error name:Error
2. Error message:BusinessError  2501000: Operation failed.
3. Error code:2501000
4. Stacktrace:
5. Cannot get SourceMap info, dump raw stack:
6. at onStart (product/phone/build/default/cache/default/default@CompileArkTS/esmodule/release/feature/systemstatus/linkspeedcomponent/src/main/ets/default/controller/NetSpeedController.ts:50:1)
7. at NetSpeedController (product/phone/build/default/cache/default/default@CompileArkTS/esmodule/release/feature/systemstatus/linkspeedcomponent/src/main/ets/default/controller/NetSpeedController.ts:43:43)

9. at func_main_0 (product/phone/build/default/cache/default/default@CompileArkTS/esmodule/release/feature/systemstatus/linkspeedcomponent/src/main/ets/default/controller/NetSpeedController.ts:325:325)
```

### 分析步骤

1. 提取日志关键信息

   通过日志信息可以确定为Error类问题，为代码主动抛出的异常。可以通过异常代码调用栈，获取错误产生位置。Cannot get SourceMap info, dump raw stack信息表示该应用为release包安装，JS栈转换eTS行列号失败，可考虑使用[堆栈轨迹分析](../harmonyos-guides/ide-release-app-stack-analysis.md)来解析行号。
2. 定位到具体代码

   通过以上 JS 堆栈，能够定位到NetSpeedController.ts文件中具体代码片段如下，异常抛出位置为wifiManager.on函数调用。

   ```
   1. wifiManager.on('wifiConnectionChange', (data) => {
   2. this.isConnected = data === 1 ? true : false;
   3. this.handleUpdateState();
   4. });
   ```

### 概述

### 分析过程

常规的JsCrash日志通过其清晰的调用堆栈，能够精准地定位到引发异常的根源代码。相比之下，OutOfMemory异常则不同，其相关的JsCrash日志往往是内存问题长期积累后的最终表现，而非一个直接的触发点。因此，OutOfMemoryError类异常JsCrash日志通常无法提供立竿见影的诊断价值，需要借助Snapshot内存占用分析工具来分析根本原因。

从上文JsCrash日志中可以看到：应用名、pid、报错信息、调用栈等，其中有参考意义的主要是报错信息，这个报错信息告诉我们，这个Crash是因为OldSpace+HugeSpace的和到达了上限，这时候尝试再往Space中分内存就发生了OOM。调用栈的参考意义不大，调用栈只说明当前分配了内存，如果到达上限，分1个字节的内存也会OOM，所以一般情况下调用栈可以忽略。

特殊情况：有些应用高频的发生OOM，而且每次的Crash文件的调用栈都一样，那说明这个调用栈可能是个高频调用，而且有泄露，最终导致了OOM，这种情况才有查看调用链的价值，其它情况只需要分析Heapsnapshot文件即可。

下面介绍两种形态如何分析：

1. 开发态Snapshot内存占用分析

   在开发态存在泄露场景，可使用DevEco Studio的Snapshot功能，应用拉起时拍摄一次内存快照，进行多次操作，操作完后退回到初始的页面，再次拍摄一次内存快照，将两次的快照进行对比，操作步骤参考[Snapshot模板基本操作](../harmonyos-guides/ide-snapshot-basic-operations.md)。
2. 运维态OOM内存占用分析

   现网：

   首先需要订阅资源泄漏事件，使用说明可参考：[订阅资源泄漏事件（ArkTS）](../harmonyos-guides/hiappevent-watcher-resourceleak-events-arkts.md)。

   当应用在ArkTS中遇到内存OOM时，系统会自动执行Heapdump，生成Rawheap文件。Rawheap的获取方法可参考：[rawheap-translator工具](../harmonyos-guides/rawheap-translator.md)。

   本地：

   * ArkTs内存管理方式和GC的时机

   首先要明确方舟虚拟机是如何管理ArkTs内存，并且在什么时候进行GC的。

   在方舟虚拟机采用的是标记清除法回收内存，内存是通过维护一棵树来管理的，这颗树的根节点被称为GC Root，ArkTs对象只要被挂载到这颗树上则说明该对象存活。很多刚接触内存的同学可能会对GC Root产生误解，认为挂在GC Root上的对象是需要被回收的，而根节点之所以叫GC Root是因为标记清楚法在标记阶段，垃圾收集器从GC Root开始遍历。所有从GC Root可达的对象都会被标记为“存活”。在清除阶段，垃圾收集器会扫描内存中的所有对象。如果某个对象没有在标记阶段被标记为存活，那么该对象就是“垃圾”，可以被回收。垃圾收集器会释放这些未标记对象所占用的内存空间。每次GC都是从根节点开始，所以根节点被称为GC Root。该树的快照如图所示：

   ![](https://contentcenter-vali-drcn.dbankcdn.cn/pvt_2/DeveloperAlliance_scene_100_1/bb/v3/7n6UIJ9rSr6VnRpYfwasUQ/zh-cn_image_0000002425122964.png?HW-CC-KV=V1&HW-CC-Date=20260429T061417Z&HW-CC-Expire=86400&HW-CC-Sign=74E162A6CD1558C7CCEB0282AEA3C840FB6BEB6CD61AB927A58C2B1F13841467)

   ArkTs的对象是存在引用和被引用关系的，图中箭头指向的都是被引用对象，所有存活的对象都直接或间接被GC Root所引用，从GC Root到被引用对象的路径为该对象的引用链。因此，解决内存泄漏的关键就是在于将对象的引用链在合适的地方（结合业务判断）断开。

## Can not get Prototype on non ECMA Object类案例

### 概述

在ArkTS与Native交互开发中，存在不当的生命周期管理操作。具体表现为，将仅在当前调用栈有效的napi\_value句柄直接存储于Native侧的全局变量中。当此句柄伴随返回值传递至ArkTS侧继续使用时，由于其原始依赖的上下文已超出生命周期且未通过napi\_ref建立持久化引用，将导致句柄失效，引发未定义行为。

该问题主要场景：

* 全局变量存储裸napi\_value
* 跨异步回调传递裸napi\_value

* 在多线程环境中共享napi\_value

[napi\_init.cpp](https://gitcode.com/HarmonyOS_Samples/BestPracticeSnippets/blob/master/JSCrash/entry/src/main/cpp/napi_init.cpp#L4-L43)

在当前实现中存在一个严重的变量生命周期管理问题。核心问题在于全局变量objValues存储的是一个napi\_value引用，该引用在Add函数中被赋值后立即失效。

具体来说，当第一次调用Add函数时：创建jsArray数组对象并将其赋值给全局变量objvalues，然后关闭局部作用域napi\_handle\_scope，这使得jsArray的引用计数减少，成为可被垃圾回收的状态。当后续调用ArkTools.hintGC()强制触发垃圾回收时，jsArray的实际内存被释放，但全局变量objValues仍然持有这个已被释放的引用地址。此后，无论是调用GetValue()函数还是再次访问objValues，都会返回一个悬空指针，这种访问已释放内存的行为导致未定义结果，通常引发程序崩溃或不可预知的内存错误。

### 修复方法

避免在NAPI中使用全局变量来存储napi\_value，改为napi\_ref来管理跨作用域的对象引用

```
1. static napi_ref objvalues = nullptr; // 使用napi_ref来管理跨作用域的对象引用
```

### 建议与总结

在ArkTS与Native交互开发中，napi\_value对象引用的生命周期管理至关重要。通过napi\_value模块接收的Native指针本质上是对原生对象的临时访问句柄，仅在当前调用栈和ArkTS对象存活期内有效。如需跨异步操作或对象生命周期持久化引用，必须使用napi\_create\_reference创建强引用，以确保垃圾回收机制识别该对象仍在被ArkTS代码持有。

若直接将裸napi\_value存储于全局变量或长生命周期对象中，短期内可能看似正常，但一旦底层原生对象被释放或ArkTS运行时执行内存回收，该引用即变为悬挂指针，导致访问异常、内存错误或数据混乱。此类问题具有隐蔽性，可能在特定触发条件下才显现，但必然在相关资源释放后导致不可预测行为。因此，开发中必须遵循：仅在同步回调内临时使用Native指针；需长期持有则必须通过napi\_ref机制实现；使用完毕后需及时释放引用；同时结合异常处理机制确保资源安全。
